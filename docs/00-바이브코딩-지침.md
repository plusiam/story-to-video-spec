# 바이브 코딩(Vibe Coding) 지침

## 개요

**바이브 코딩**이란 복잡한 코딩 과정 없이 AI와의 프롬프트 대화를 통해 빠르게 애플리케이션을 개발하는 방식이다.

이 문서는 "스토리 구성 웹학습지" 프로젝트에서 바이브 코딩 방식을 효과적으로 활용하기 위한 지침을 정의한다.

---

## 1. 기획 지침: '바이브'와 '결과물' 중심

### 1.1 전통적 방식 vs 바이브 코딩

| 분류 | 전통적 방식 | 바이브 코딩 |
|------|-------------|-------------|
| 핵심 기획 | 상세한 기능 정의, 기술 스택 선정 | **바이브(Vibe) 정의**: 제품이 사용자에게 어떤 느낌과 경험을 줄 것인가? |
| 요구 사항 | 기능 목록 (Functional Requirements) | **결과물(Outcome) 정의**: 사용자가 무엇을 할 수 있게 되는가? |
| 프로토타입 | 와이어프레임, 상세 설계 | **싱글 프롬프트 계획**: 핵심 기능을 한 번에 구현하기 위한 최적의 프롬프트 |

### 1.2 바이브 정의 예시

이 프로젝트의 바이브:

```
🎯 프로젝트 바이브

[교사 경험]
- "복잡한 설정 없이 바로 시작하는 느낌"
- "학생들의 창작 활동을 한눈에 보는 뿌듯함"
- "기술에 능숙하지 않아도 자신감을 주는 친절함"

[학생 경험]
- "게임처럼 재미있는 스토리 만들기"
- "내가 만든 작품이 멋지게 보이는 성취감"
- "선생님과 친구들에게 보여주고 싶은 자랑스러움"
```

### 1.3 결과물(Outcome) 정의 방식

**❌ 기능 중심 (전통적)**
```
- 학생 로그인 기능 구현
- PIN 해시 저장 기능
- QR 코드 생성 기능
```

**✅ 결과물 중심 (바이브 코딩)**
```
- "학생이 이름과 번호만 입력하면 바로 자기 작품 페이지에 들어간다"
- "교사가 QR 코드를 교실에 붙여두면 학생들이 스마트폰으로 바로 접속한다"
- "처음 온 학생도 3번의 탭만으로 첫 작품을 시작할 수 있다"
```

## 2. 개발 지침: 프롬프트 엔지니어링

### 2.1 프롬프트 작성 원칙

| 원칙 | 설명 | 예시 |
|------|------|------|
| **명료성** | 애매모호한 표현 대신 구체적으로 | ❌ "로그인 기능 만들어줘" → ✅ "이름+번호+6자리PIN으로 로그인하는 GAS 함수, 실패 시 한글 에러 메시지 반환" |
| **맥락 제공** | 프로젝트 배경과 제약 조건 포함 | "초등학생 대상, 구글 스프레드시트를 DB로 사용, 모바일 우선" |
| **단계적 접근** | 큰 기능을 작은 단위로 분해 | 1) 학생 찾기 함수 → 2) PIN 검증 함수 → 3) 로그인 통합 |
| **예시 포함** | 입력/출력 예시 제공 | "입력: {name: '홍길동', number: 15}, 출력: {success: true, token: 'tk_xxx'}" |

### 2.2 효과적인 프롬프트 템플릿

```
[목표]
{무엇을 만들 것인지 한 문장으로}

[맥락]
- 대상: {누가 사용하는가}
- 환경: {어떤 플랫폼/기술}
- 제약: {지켜야 할 조건}

[세부 요구사항]
1. {구체적 요구사항 1}
2. {구체적 요구사항 2}
3. {구체적 요구사항 3}

[입출력 예시]
입력: {예시}
출력: {예시}

[참고]
- 기존 코드: {있다면 제공}
- 스타일: {코딩 스타일 가이드}
```

### 2.3 프롬프트 예시 (이 프로젝트)

```
[목표]
학생이 QR 코드를 스캔하면 바로 로그인되는 함수 만들기

[맥락]
- 대상: 초등학생 (4~6학년)
- 환경: Google Apps Script, 스프레드시트 DB
- 제약: 외부 라이브러리 없이 GAS 내장 기능만 사용

[세부 요구사항]
1. URL 파라미터에서 token 값 추출
2. STUDENTS 시트에서 해당 토큰의 학생 찾기
3. 학생 상태가 'active'인 경우만 로그인 성공
4. 마지막 접속 시간 업데이트
5. 한글 에러 메시지 반환

[입출력 예시]
입력: token = "tk_abc123xyz"
성공: { success: true, name: "홍길동", number: 15 }
실패: { success: false, error: "유효하지 않은 QR입니다." }

[참고]
- STUDENTS 시트 컬럼: 이름, 번호, PIN해시, 토큰, 등록일, 마지막접속, 상태
- 기존 findStudentByToken 함수 있음 (02-DB스키마.md 참조)
```

### 2.4 반복 및 검증 프로세스

```
[바이브 코딩 개발 사이클]

1️⃣ 프롬프트 작성
   ↓
2️⃣ AI 응답 확인
   ↓
3️⃣ 즉시 테스트 (GAS 편집기에서 실행)
   ↓
4️⃣ 결과 확인
   │
   ├─ ✅ 성공 → 다음 기능으로
   │
   └─ ❌ 실패 → 프롬프트 수정 후 재시도
                (코드 수정 X, 프롬프트 수정 O)
```

### 2.5 디버깅 기준: 프롬프트 vs 코드

| 상황 | 접근 방식 |
|------|----------|
| 로직 오류 | 프롬프트 수정하여 새 코드 생성 |
| 오타/문법 오류 | 프롬프트에 "문법 오류 수정" 요청 |
| 부분 수정 필요 | 특정 부분만 지정하여 수정 요청 |
| AI 한계 도달 | 직접 코드 수정 (최후의 수단) |

**원칙: 프롬프트 튜닝으로 해결 불가능할 때만 직접 코드 편집**

## 3. 프론트엔드 디자인 지침

### 3.1 UI 바이브 정의

이 프로젝트의 UI 바이브:

```
🎨 UI 바이브 키워드

- 친근한 (Friendly): 초등학생이 무섭지 않게
- 밝은 (Bright): 밝은 색상, 긍정적 분위기
- 단순한 (Simple): 한 화면에 한 가지 일만
- 격려하는 (Encouraging): 칭찬과 응원 메시지
- 재미있는 (Playful): 약간의 이모지와 애니메이션
```

### 3.2 스타일 프롬프트 통일

모든 UI 생성 프롬프트에 포함할 스타일 키워드:

```
[UI 스타일 가이드]
- 색상: 파란색(#4A90D9) 기본, 녹색(#7CB342) 강조
- 글꼴: Noto Sans KR, 16px 이상
- 버튼: 둥근 모서리(8px), 큰 터치 영역(44px 이상)
- 간격: 여백 충분히, 요소 간 16px 이상
- 피드백: 모든 동작에 즉각적 반응 (로딩, 성공, 실패)
```

### 3.3 UI 프롬프트 예시

```
[목표]
학생 로그인 화면 HTML 만들기

[UI 바이브]
- 초등학생 대상, 친근하고 밝은 느낌
- 모바일 우선, 세로 화면에 최적화
- 입력 필드는 크고 명확하게

[구성 요소]
1. 상단: 선생님 이름 + 학교/학급 표시
2. 중앙: 이름 입력, 번호 입력, PIN 입력 (각각 별도 필드)
3. 하단: 로그인 버튼 (크고 눈에 띄게)
4. 에러: 빨간색 배경 + 아이콘으로 명확하게

[스타일]
- 배경: 밝은 회색 (#F5F5F5)
- 카드: 흰색, 그림자, 둥근 모서리
- 버튼: 파란색 (#4A90D9), 흰색 글자
- 이모지: 적절히 사용 (📚, 👋, 🔑)
```

### 3.4 사용자 경험 중심 프롬프트

**❌ 기능 중심**
```
"로그인 폼 만들어줘"
```

**✅ 경험 중심**
```
"처음 온 학생이 '와, 이거 쉽네!' 하며 30초 안에
자기 이름과 번호를 입력하고 로그인하는 화면"
```

## 4. 백엔드 디자인 지침

### 4.1 데이터 흐름 명확화

모든 기능에 대해 입력-처리-출력을 먼저 정의:

```
[기능: 학생 로그인]

입력 (Input)
├── 이름: 문자열
├── 번호: 숫자
└── PIN: 6자리 숫자

처리 (Process)
├── 1. STUDENTS 시트에서 학생 찾기
├── 2. PIN 해시 비교
├── 3. 상태 확인 (active만 허용)
└── 4. 마지막 접속 시간 업데이트

출력 (Output)
├── 성공: { success: true, name, number }
└── 실패: { success: false, error: "메시지" }
```

### 4.2 API 설계 프롬프트

```
[목표]
학생 작품 저장 API 함수 만들기

[데이터 흐름]
입력: { studentName, studentNumber, workData(JSON), step(1/2/3) }
처리: WORKS_STEP{N} 시트에 저장/업데이트
출력: { success, workId, savedAt }

[제약 조건]
- 같은 학생+단계의 기존 작품이 있으면 업데이트
- 없으면 새로 생성
- workData는 JSON 문자열로 저장
- 생성일/수정일 자동 기록

[에러 케이스]
- 학생 없음: "등록되지 않은 학생입니다"
- 잘못된 step: "올바른 단계를 선택해주세요"
- 데이터 오류: "작품 데이터가 올바르지 않습니다"
```

### 4.3 보안 고려사항

바이브 코딩에서도 보안은 직접 검토:

| 영역 | AI 생성 | 직접 검토 필요 |
|------|:-------:|:--------------:|
| 기본 CRUD 로직 | ✅ | - |
| 입력값 검증 | ✅ | ✅ |
| PIN 해시 처리 | ✅ | ✅ |
| 권한 확인 로직 | ✅ | ✅ |
| 에러 메시지 (정보 노출) | ✅ | ✅ |

**원칙: AI가 생성한 보안 관련 코드는 반드시 직접 검토**

├── 📄 인증.prompt.md
│   ├── PIN 로그인

│
└── 📄 UI-컴포넌트.prompt.md
    ├── 로그인 화면
    ├── 대시보드
    └── 작품 편집기

### 5.3 AI 한계 대응

바이브 코딩으로 해결하기 어려운 경우:

| 상황 | 대응 방안 |
|------|----------|
| 복잡한 비즈니스 로직 | 작은 단위로 분해 후 조합 |
| 외부 API 연동 | 공식 문서 참조하여 프롬프트에 포함 |
| 성능 최적화 | 기본 동작 확인 후 별도 최적화 |
| 특수한 라이브러리 | 해당 라이브러리 문서 프롬프트에 포함 |
| AI 반복 오류 | 직접 코딩으로 전환 (최후 수단) |

이 프로젝트에서 바이브 코딩으로 구현할 때 반드시 따라야 할 협의 결과:

| # | 주제 | 핵심 결정 | 프롬프트 키워드 |
|---|------|----------|----------------|

| 4 | QR 코드 | 반QR + 학생별QR(바로입장) | "토큰 포함 URL로 PIN 없이 입장" |

| 9 | 교사 설정 | 하이브리드 (웹+시트) | "초기설정은 웹, 일괄작업은 시트" |
