## 개요

점조직 모델 기반 시스템의 세부 구현 사항에 대한 결정 내용을 정리한다.

## 주제 1: 원클릭 설치 구현

### 결정 사항

| 항목 | 결정 | 이유 |
|------|------|------|
| **구현 방식** | 하이브리드 (B로 시작) | 빠른 출시 + 점진적 개선 |
| **템플릿 위치** | 둘 다 (공개 스프레드시트 + GitHub) | 유연성 확보 |
| **초기 설정** | 나중에 입력 가능 | 빠른 체험 우선 |

### 구현 로드맵

```
[1단계] 템플릿+1클릭배포 방식 (B)
1. "시스템 만들기" 클릭
2. 구글 로그인
3. 템플릿이 내 드라이브에 자동 복사
4. 안내: "배포 버튼을 눌러주세요"
5. 교사가 배포 클릭 (1회)
6. URL 생성 완료

[2단계] 사용자 피드백 수집 (1-2개월)

[3단계] 필요시 완전자동 방식 (A)으로 개선
```

| 항목 | 결정 | 이유 |
|------|------|------|
| **버전 체크 방식** | GitHub 체크 | 무료, 안정적, 이미 사용 중 |
| **버전 번호 체계** | 시맨틱 (2.1.3) | 업계 표준, 변경 규모 파악 쉬움 |

## 주제 4: QR 코드 생성

### 결정 사항

| 항목 | 결정 | 이유 |
|------|------|------|
| **생성 방식** | 자바스크립트 라이브러리 | 외부 의존 없음, 오프라인 가능 |
| **반 전체 QR** | 유지 | 신규 가입, 집에서 접속용 |
| **학생별 QR** | 추가 | 교실에서 바로 입장 |
| **학생별 QR 인증** | 영구 토큰 (PIN 불필요) | QR 스캔 = 인증 완료 |
| **다운로드 형식** | PNG + PDF (4명/페이지) + ZIP | 상황별 선택 가능 |
| **QR 재발급** | 제공 | 필요시 토큰 갱신 |

### 인증 흐름

```
[QR 스캔 시 - 교실]
학생 QR 스캔 → 토큰 확인 → 바로 입장 ✅

[URL 직접 접속 시 - 집]

```

## 주제 6: PIN 보안 정책

### 결정 사항

| 항목 | 결정 | 이유 |
|------|------|------|
| **해시 알고리즘** | SHA-256 | GAS 기본 지원, 충분한 보안 |
| **Salt 관리** | 시스템 공통 Salt | 단순함, 초등학생 대상 적합 |
| **PIN 길이** | 6자리 | 보안과 편의 균형 |
| **문자 종류** | 숫자만 | 입력 간편 |
| **변경 허용** | 교사만 재설정 | 관리 편의 |
| **실패 제한** | 없음 | 학생 잠금 시 교사 부담 방지 |

### 해시 구현

```javascript
function hashPin(pin) {
  const salt = getSalt(); // SETTINGS에서 가져옴
  const salted = salt + pin;
  const hash = Utilities.computeDigest(
    Utilities.DigestAlgorithm.SHA_256,
    salted
  );
  return Utilities.base64Encode(hash);
}

function getSalt() {
  const sheet = SpreadsheetApp.getActive().getSheetByName('SETTINGS');
  const data = sheet.getDataRange().getValues();
  for (let i = 0; i < data.length; i++) {
    if (data[i][0] === 'pinSalt') {
      return data[i][1];
    }
  }
  // Salt 없으면 생성
  const newSalt = Utilities.getUuid();
  sheet.appendRow(['pinSalt', newSalt]);
  return newSalt;
}
```

### PIN 재설정 (교사용)

```javascript
function resetStudentPin(name, number, newPin) {
  const sheet = SpreadsheetApp.getActive().getSheetByName('STUDENTS');
  const data = sheet.getDataRange().getValues();

  for (let i = 1; i < data.length; i++) {
    if (data[i][0] === name && data[i][1] === number) {
      const newHash = hashPin(newPin);
      sheet.getRange(i + 1, 3).setValue(newHash); // PIN해시 컬럼
      return { success: true };
    }
  }
  return { success: false, error: '학생을 찾을 수 없습니다.' };
}
```

## 주제 7: 학생 가입 정책

### 결정 사항

| 항목 | 결정 | 이유 |
|------|------|------|
| **가입 방식** | 교사 선등록 필수 | 충돌 방지, 보안 강화 |
| **교사 등록 방식** | 개별 + 일괄 + 엑셀 모두 제공 | 상황별 선택 |
| **PIN 설정 주체** | 교사 선택 (학생 직접 or 교사 지정) | 유연성 |
| **학생 상태** | pending → active → inactive | 명확한 상태 관리 |

### 교사 선등록 방식 채택 이유

**기존 문제:**
```
학생 자유 가입 시 충돌 가능성
- 학생 A가 먼저 가입 (이름+번호+PIN)
- 교사가 나중에 같은 학생 등록 시도
- 데이터 충돌 발생
```

**해결:**
```
교사가 먼저 학생 명단 등록 (이름+번호)
→ 학생은 등록된 경우에만 PIN 설정 가능
→ 충돌 원천 차단
```

### 학생 상태

| 상태 | 설명 | PIN | QR |
|------|------|-----|-----|
| pending | 교사 등록, PIN 미설정 | 없음 | ❌ |
| active | PIN 설정 완료 | 있음 | ✅ |
| inactive | 교사가 비활성화 | 유지 | ❌ |

### 가입 흐름

**경로 A: 학생이 PIN 설정 (권장)**
```
[교사]
1. 학생 명단 등록 (이름+번호만)
2. PIN 설정: "학생이 직접"
3. 상태: pending

[학생]
1. URL 접속
2. 이름+번호 입력
3. "처음 오셨네요! PIN을 만들어주세요"
4. PIN 6자리 설정
5. 상태: active → QR 사용 가능
```

**경로 B: 교사가 PIN 설정**
```
[교사]
1. 학생 명단 등록 (이름+번호+PIN)
2. 상태: active (바로 사용 가능)
3. QR 즉시 배포 가능

[학생]
1. QR 스캔 → 바로 입장
   또는
2. URL 접속 → 이름+번호+PIN → 입장
```

### 미등록 학생 처리

```
학생이 접속 시도 → 등록 여부 확인
  ├─ 등록됨 (pending) → PIN 설정 화면
  ├─ 등록됨 (active) → 로그인 화면
  └─ 미등록 → "선생님께 문의하세요" 안내
```

## 주제 8: 프론트엔드 아키텍처

### 결정 사항

| 항목 | 결정 | 이유 |
|------|------|------|
| **아키텍처** | GAS 다중 파일 (Option C) | 점조직 독립성 유지, 사본 복사 시 완전 포함 |
| **파일 분리** | 기능별 HTML 파일 분리 | 유지보수 용이, 코드 가독성 |
| **파일 포함 방식** | GAS include 패턴 | 표준 방식, 안정적 |

### 검토한 옵션들

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A. 단일 파일 | 모든 코드를 Index.html에 | 단순함 | 유지보수 어려움, 파일 비대 |
| B. 외부 분리 | GitHub Pages + GAS API | 프론트 자유도 | 점조직 독립성 훼손, CORS |
| **C. GAS 다중 파일** | GAS 내 여러 .html 파일 | 독립성 유지, 관리 용이 | 약간의 학습 필요 |
| D. 하이브리드 | CDN + GAS | 최신 라이브러리 | 외부 의존성 |

### Option C 채택 이유

```
1. 점조직 독립성 유지
   - 사본 복사 시 모든 파일 자동 포함
   - 외부 서버 의존 없음
   - 각 교사 시스템 완전 독립

2. 유지보수 용이
   - 기능별 파일 분리로 수정 범위 명확
   - 팀 작업 시 충돌 최소화
   - 코드 재사용 가능

3. GAS 표준 방식
   - 공식 지원 패턴
   - 안정적 동작 보장
   - 풍부한 참고 자료
```

```html
        return;
      }

      // 서버에서 해당 페이지 HTML 가져오기
      const html = await callServer('getPage', template, params);
      document.getElementById('app').innerHTML = html;

      // 페이지 초기화 함수 호출
      if (window[`init${template}`]) {
        window[`init${template}`](params);
      }
    }
  };

  // 초기 라우팅
  document.addEventListener('DOMContentLoaded', () => {
    const params = new URLSearchParams(window.location.search);
    const token = params.get('token');

    if (token) {
      // QR 토큰으로 접속
      Router.navigate('student-main', { token });
    } else {
      // 일반 접속 → 로그인
      Router.navigate('student-login');
    }
  });
</script>
```

### 페이지 로드 함수 (Code.gs)

```javascript
function getPage(templateName, params) {
  const template = HtmlService.createTemplateFromFile(templateName);

  // 파라미터 전달
  template.params = params || {};

  return template.evaluate().getContent();
}
```

### 결정 사항

### 검토한 옵션들

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A. 웹 UI만 | 모든 설정을 웹에서 | 친절한 UX, 실수 방지 | 개발 공수 큼 |
| B. 시트 직접만 | 스프레드시트에서 직접 | 개발 최소화 | 첫인상 부담, 실수 위험 |
| **C. 하이브리드** | 핵심은 웹 + 고급은 시트 | 균형 잡힌 접근 | 둘 다 관리 필요 |

### Option C 채택 이유

```
1. 첫인상 관리
   - 첫 설치 시 웹 UI로 친절하게 안내
   - "시트 열어서 A열에..." 부담 제거
   - 이탈률 최소화

2. 파워유저 지원
   - 엑셀 능숙한 교사는 시트에서 효율적 작업
   - 30명 학생 복붙 등 일괄 작업 가능
   - 데이터 직접 확인/수정 가능

3. 개발 효율성
   - 모든 기능 웹 UI화 불필요
   - 핵심 기능만 웹으로 제공
   - 유지보수 부담 감소
```

```javascript
function protectHeaders() {
  const ss = SpreadsheetApp.getActive();
  const sheets = ['STUDENTS', 'WORKS_STEP1', 'WORKS_STEP2', 'WORKS_STEP3', 'SETTINGS'];

  sheets.forEach(sheetName => {
    const sheet = ss.getSheetByName(sheetName);
    if (sheet) {
      const protection = sheet.getRange(1, 1, 1, sheet.getLastColumn())
        .protect()
        .setDescription('헤더 보호 - 수정하지 마세요');

      // 소유자만 수정 가능
      protection.removeEditors(protection.getEditors());
      if (protection.canDomainEdit()) {
        protection.setDomainEdit(false);
      }
    }
  });
}
```

```javascript
function validateHeaders() {
  const expected = {
    'STUDENTS': ['이름', '번호', 'PIN해시', '토큰', '등록일', '마지막접속', '상태'],
    'WORKS_STEP1': ['학생이름', '학생번호', '작품데이터', '생성일', '수정일', '완료여부', '상태']
    // ...
  };

  Object.keys(expected).forEach(sheetName => {
    const sheet = SpreadsheetApp.getActive().getSheetByName(sheetName);
    if (sheet) {
      const headers = sheet.getRange(1, 1, 1, expected[sheetName].length).getValues()[0];
      const isValid = expected[sheetName].every((h, i) => headers[i] === h);

      if (!isValid) {
        SpreadsheetApp.getUi().alert(
          `⚠️ ${sheetName} 시트의 헤더가 변경되었습니다.\n` +
          `시스템이 정상 작동하지 않을 수 있습니다.\n` +
          `헤더를 복원하거나 관리자에게 문의하세요.`
        );
      }
    }
  });
}
```
